# Выбор образа для Docker-установки: `latest` или `dev`

> **ВНИМАНИЕ — перед обновлением живой установки.** `docker compose pull` тянет тот тег, что прописан в `docker-compose.yml`/`.env` **прямо сейчас**. Прежде чем обновлять, проверьте тег: для стабильной версии он должен быть `latest`. Если там `dev` — обновление утянет тестовую сборку.

Этот документ — про то, **какой образ тянет ваша Docker-установка** и как переключаться между стабильной и тестовой версией, не поймав тестовую сборку случайно.

## Какие теги публикуются

Образ лежит в GHCR: `ghcr.io/mrvibecodic/remnawave-subscription-middleware`.

| Тег | Что это | Кому |
|---|---|---|
| `latest` | **Стабильный.** Собирается только из ветки `main`. | Обычная установка (по умолчанию). |
| `dev` | **Тестовый.** Собирается из ветки `dev`, может содержать недоделанное и ломаться. | Тем, кто осознанно тестирует новое. |
| `main` | То же содержимое, что `latest` (тег по имени стабильной ветки). | Совместимость. |
| `v1.2.3` | Конкретный релиз. | Если нужна фиксированная версия. |
| `sha-…` | Конкретный коммит. | Отладка/откат на точную сборку. |

По умолчанию ставится **`latest`**.

## Где меняется тег

**Если использовали `docker-compose.example.yml`** — строка образа параметризована:

```yaml
image: ghcr.io/mrvibecodic/remnawave-subscription-middleware:${SUBMW_IMAGE_TAG:-latest}
```

Тег берётся из переменной `SUBMW_IMAGE_TAG`. Если её не задать — будет `latest`. Задать можно в файле `.env` рядом с `docker-compose.yml`:

```dotenv
SUBMW_IMAGE_TAG=latest
```

**Если ставили через `install.sh` (режим «рядом с панелью — Docker»)** — установщик спрашивает тег при установке (по умолчанию `latest`) и пишет его прямо в строку `image:` готового `docker-compose.yml`. Менять — там же.

## Переключиться на тестовую сборку (`dev`) и обратно

```bash
cd /opt/remnawave-subscription-middleware     # каталог с docker-compose.yml
# поставить dev: в .env  SUBMW_IMAGE_TAG=dev   (или поправить строку image: на :dev)
docker compose pull && docker compose up -d
```

Вернуться на стабильную — выставить тег обратно в `latest` и снова `docker compose pull && docker compose up -d`.

## Важно при обновлении

`docker compose pull` тянет **тот тег, что прописан в compose прямо сейчас** — он не «знает», что вы хотели стабильную версию.

- Хотите остаться на стабильной → **перед** `docker compose pull` убедитесь, что тег = `latest` (не `dev`).
- Хотите протестировать новое → временно поставьте `dev`, проверьте, потом верните `latest`.

`config.php` и база данных живут в volume `submw-data` и переживают любую смену тега и `docker compose pull`.
